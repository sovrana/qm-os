# Framework Development & Problem-Solving

## Framework Development Process

**Pattern Recognition:**
When you notice repeating strategic patterns across contexts:
1. **Name it** - Create sticky label
2. **Formalise it** - Document as reusable framework
3. **Test it** - Apply to next engagement
4. **Refine it** - Iterate based on application learnings
5. **Scale it** - Framework becomes standard playbook

## Problem-Solving Patterns

**Political Architecture:**
When organisational tensions block progress, don't reorganise - architect the solution:
- **Internal outsourcer model** - Solves partner resistance (who keeps the client relationship?)
- **Transfer pricing** - Solves cost allocation disputes (who pays for QC?)
- **Federated model** - Solves centralisation resistance (central standards, BU ownership)
- Pattern: Political deadlock -> system design that makes tension irrelevant

**Pre-mortem Discipline:**
Before proposing, ask "How does this fail?":
- Transformation failure modes (Shadow-AI, Ivory-Tower COE, Culture Shock)
- Over-claiming risk (for credibility)
- Exception handling (what breaks the model?)
- Pattern: Identify attack surface -> design countermeasures -> present defensible plan

**Stage-Gating for Credibility:**
Structure investments to build conviction incrementally:
- Capped exposure at early gates
- Explicit kill-rate >60%
- Benefit-to-burn ratio >1.5 each quarter
- Sensitivity analysis (3 scenarios minimum)
- Pattern: Big ambition -> small gates -> prove or kill fast
