# The Gearbox Framework: Three Operating Modes for AI Agent Deployment

## Core Thesis

Professional services firms face a binary choice: bolt AI onto existing workflows (incremental) or redesign the work itself (transformational). The Gearbox Framework rejects this binary. Instead, it proposes three operating modes that coexist within the same system, shifting work between modes as the system learns.

## The Three Gears

**Instrument (1st Gear):** The system observes. Humans do all work. Agents monitor patterns, flag anomalies, surface insights. No workflow changes. The economics are visibility - you see what was previously invisible.

**Prepare (2nd Gear):** Agents execute assembly and preparation steps - 40-70% of workflow by volume. Humans own judgment and sign-off. The system learns which questions matter. Margin improvement: +5-10 percentage points.

**Execute (3rd Gear):** Agents own 80-95% of the workflow. Humans govern exceptions. Supervisor ratio shifts to 1:20+. The system learns answers, not just questions. Margin improvement: +15-22 percentage points.

## Key Claims

1. Work naturally shifts up through gears over time. The first due diligence in a new sector starts in Prepare. The twentieth runs 60% in Execute.

2. The margin gap between Prepare and Execute exists because the judgment portion of work has a cost floor that only drops when the system proves it can replicate judgment reliably.

3. The gearbox is a discovery mechanism. It reveals which "judgment work" is actually pattern recognition that has never been measured.

4. Most firms attempting AI transformation skip Instrument entirely and jump to Execute. This fails because the system has no learned patterns to draw from.

## Proposed Adoption Path

Phase 1 (months 1-3): Deploy in Instrument across three service lines. Capture baseline metrics.
Phase 2 (months 4-8): Shift highest-volume workflows to Prepare. Measure margin impact.
Phase 3 (months 9-18): Graduate proven workflows to Execute. Scale horizontally.

## Open Questions

- How do you handle workflows that oscillate between gears depending on client complexity?
- What governance prevents premature promotion to Execute before the system has sufficient learned patterns?
- Is the 1:20+ supervisor ratio realistic outside highly standardised workflows?
